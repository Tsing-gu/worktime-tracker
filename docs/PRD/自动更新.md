# 自动更新 PRD

## 1. 背景与目标

当前通过 DMG 手动分发，用户需主动询问/下载新版本，存在：
- 用户不知道有新版
- 手动下载、拖拽覆盖流程繁琐
- 发版后老版本无法回收

**目标**：引入 Sparkle 框架实现 macOS 桌面应用自动更新，用户启动应用后自动检查、下载、安装新版本。

**约束**：
- 无 Apple Developer ID（未签名），接受 Gatekeeper 首次拦截（用户右键打开即可）
- 小范围内部使用（数十用户），无需灰度/分渠道
- 零服务器成本，使用 GitHub Releases 托管更新包

## 2. 方案选型

采用 **Sparkle**（macOS 事实标准更新框架）+ **GitHub Releases** 托管：

| 项 | 选择 | 理由 |
|---|---|---|
| 更新框架 | Sparkle 2.x | macOS 原生、增量更新、开源 MIT、零费用 |
| 代码签名 | 无 | 未签名也能跑 Sparkle，仅 Gatekeeper 首次拦截 |
| 更新包格式 | DMG | 与现有分发一致 |
| 版本清单托管 | GitHub Releases raw URL | 免费、自带 CDN |
| EdDSA 签名 | 启用 | Sparkle 自带，防篡改（非 Apple 签名） |
| 集成方式 | pyobjc-framework-Sparkle | PySide6/Python 桥接 |

## 3. 架构与数据流

### 3.1 角色与组件

```
┌─────────────┐   ①拉取 appcast.xml   ┌──────────────────┐
│  客户端 App  │ ──────────────────────> │ GitHub Releases  │
│ (内嵌 Sparkle)│ <────────────────────── │ (appcast + DMG)  │
└─────────────┘   ②返回版本清单+DMG URL   └──────────────────┘
       │
       │ ③对比版本号 → 有新版 → 下载 DMG → 校验 EdDSA 签名
       │ ④退出主进程 → 解压替换 .app → 重启
       ▼
┌─────────────────────┐
│ /Applications/工时   │
│ 计算器.app (新版本)  │
└─────────────────────┘
```

### 3.2 发版流程（开发者侧）

1. 修改代码 → 更新 VERSION 文件 → 更新 CHANGELOG.md
2. `pyinstaller 工时计算器.spec` 打包 → 生成 dist/工时计算器.app
3. `hdiutil` 生成 DMG（沿用现有流程）
4. 用 Sparkle 的 `sign_update` 工具对 DMG 生成 EdDSA 签名
5. 在 GitHub 创建 Release（tag = v版本号），上传 DMG 为附件
6. 更新仓库根目录 appcast.xml（新增一条 item，填 DMG 的 GitHub Release 下载 URL + 签名 + 长度）
7. 提交推送 appcast.xml（Sparkle 通过 raw.githubusercontent.com 拉取）

### 3.3 更新流程（用户侧，全自动）

1. 用户启动 v1.0
2. Sparkle 后台异步拉取 appcast.xml
3. 对比 sparkle:version，若服务器版本 > 本地版本：
   - 弹窗显示新版号 + 更新说明 + "立即更新/稍后/跳过该版"按钮
4. 用户点"立即更新"：
   - Sparkle 下载 DMG 到临时目录
   - 校验 EdDSA 签名（防篡改）
   - 弹"是否安装并重启"确认
   - 退出主进程 → 挂载 DMG → 用新 .app 替换 /Applications/工时计算器.app
   - 重新启动新版本
5. Gatekeeper 首次拦截 → 用户右键打开 → 后续正常

### 3.4 首次分发

- 第一版 v1.0 由开发者手动发 DMG 给用户安装
- 该 DMG 内已嵌入 Sparkle，配置好 feedURL 指向 GitHub appcast.xml
- 用户安装后即具备自动更新能力，后续不再需要手动下载

## 4. 实现改动点

### 4.1 打包配置（工时计算器.spec）

```python
# 新增/修改
a = Analysis(
    ['main.py'],
    ...
    hiddenimports=['openpyxl', 'Sparkle'],  # 新增 Sparkle
)

app = BUNDLE(
    coll,
    name='工时计算器.app',
    bundle_identifier='com.tsinggu.worktime-tracker',  # 新增
    info_plist={
        'CFBundleIdentifier': 'com.tsinggu.worktime-tracker',
        'CFBundleShortVersionString': '1.0.0',  # 从 VERSION 读取
        'SUFeedURL': 'https://raw.githubusercontent.com/用户名/仓库/main/appcast.xml',
        'SUEnableAutomaticChecks': True,
        'SUAutomaticallyUpdate': False,  # 手动确认后自动安装
    },
)
```

Sparkle.framework 放入 `Frameworks/`，spec 配置 `frameworks`/`binaries` 嵌入。

### 4.2 启动初始化（main.py，新增约 15 行）

```python
try:
    from Sparkle import SUUpdater
    updater = SUUpdater.sharedUpdater()
    updater.checkForUpdatesInBackground()  # 启动时后台检查
except ImportError:
    pass  # 开发环境无 Sparkle 时静默跳过
```

业务代码（src/ 下 tracker/calculator/database/ui）**完全不动**。

### 4.3 依赖

`requirements.txt` 新增：
```
pyobjc-framework-Sparkle
```

### 4.4 新增文件

- `appcast.xml`（仓库根目录）：版本清单，发版时更新
- `scripts/sign_update.sh`（可选）：封装 sign_update 命令
- `docs/PRD/自动更新.md`（本文件）

## 5. appcast.xml 模板

```xml
<?xml version="1.0" standalone="yes"?>
<rss version="2.0" xmlns:sparkle="http://www.andymatuschak.org/xml-namespaces/sparkle">
  <channel>
    <title>工时计算器更新</title>
    <item>
      <title>版本 1.0.0</title>
      <pubDate>初始版本</pubDate>
      <sparkle:version>1</sparkle:version>
      <sparkle:shortVersionString>1.0.0</sparkle:shortVersionString>
      <description>初始发布</description>
      <enclosure
        url="https://github.com/用户名/仓库/releases/download/v1.0.0/工时计算器.dmg"
        sparkle:edSignature="签名串"
        length="DMG文件字节数"
        type="application/octet-stream"/>
    </item>
  </channel>
</rss>
```

## 6. 发版 SOP（标准操作流程）

```
1. 改代码 → 更新 VERSION → 写 CHANGELOG
2. python -m PyInstaller 工时计算器.spec
3. 生成 DMG（沿用现有流程）
4. ./bin/sign_update ./dist/工时计算器.dmg
   → 输出: sparkle:edSignature="xxx" length="12345"
5. git tag v版本号 && git push --tags
6. GitHub 上基于该 tag 创建 Release，上传 DMG
7. 编辑 appcast.xml 新增 item（粘贴签名串、length、DMG URL、版本号）
8. git add appcast.xml && git commit && git push
9. 用户下次启动 app 自动检测到新版
```

## 7. 风险与权衡

| 风险 | 影响 | 缓解 |
|---|---|---|
| 未签名导致 Gatekeeper 拦截 | 每次更新后首次打开需右键放行 | 在 README/首次启动弹窗说明操作；小范围可接受 |
| GitHub raw URL 国内访问慢 | 检查更新偶尔超时 | Sparkle 自带重试；可后续换 Gitee 镜像或 OSS |
| pyobjc-framework-Sparkle 兼容性 | 可能与 PySide6 打包冲突 | 集成时验证 PyInstaller 能正确打包 framework |
| 用户首次分发后 feedURL 写死 | 后续无法更改检查地址 | 首次配置前确认 GitHub 仓库路径不变；或通过 settings 提供 feedURL 覆盖 |

## 8. 不在本期范围

- 代码签名（需 $99/年 Developer ID，后续视用户量决定）
- 增量更新（Sparkle 自带支持，但未签名下不可用 delta，全量 DMG 即可）
- 多渠道/灰度发布
- 自动更新开关的 UI 设置项（Sparkle 默认行为已足够）

## 9. 验收标准

- [ ] 打包后 .app 内含 Sparkle.framework
- [ ] 启动应用后 5 秒内后台请求 appcast.xml
- [ ] 有新版时弹出 Sparkle 标准更新弹窗
- [ ] 点击"立即更新"能完成下载、替换、重启
- [ ] 更新后版本号正确显示为新版
- [ ] 首次分发 DMG 给用户安装后，具备自动更新能力
